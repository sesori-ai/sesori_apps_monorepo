Plugin Lifecycle Migration Plan

Move all OpenCode runtime ownership (spawn, ports, health, ownership records, DB maintenance) out of bridge/app and into the OpenCode plugin, behind a plugin interface that supports managed-runtime, direct-CLI, and remote-server plugins — with per-plugin enable/start/stop. Delivered as a ladder of small, always-green PRs.
Status: in progress (rev 4 — PRs 1–8 merged (PR 8 = #239), PR 9 shipped #243; shipped cards carry delta callouts binding on later PRs) Date: 2026-06-10 (rev 3: 2026-06-11; rev 4: 2026-06-12) PRs: 15, strictly sequential Max PR size: ~2,000 changed lines (hard) Branch base: main

0Ground rules

  • Always green. After every PR: dart analyze --fatal-infos clean in every touched package (this is what bridge-ci.yml runs — note: a cross-package @Deprecated fails it, see PR 2), all tests pass, and the bridge runs end-to-end exactly as before (start, login, OpenCode spawn, mobile session flow, --no-auto-start attach mode, clean shutdown). No PR depends on a future PR to compile or behave.
  • No breaking changes. CLI flags (--port, --no-auto-start, --password, --opencode-bin) keep their exact names and semantics throughout — headless/systemd units (docs/SETUP_HEADLESS_VM.md) must never need editing. The on-disk ownership file <cacheDir>/runtime/opencode-processes.json keeps its exact schema and steady-state bytes forever (frozen contract — self-updating fleets mix bridge versions). Anything new the supervisor needs to persist goes in separate, new files that old bridges never read.
  • ≤ ~2,000 changed lines per PR, as git diff --stat counts them — moves count on both sides, generated files count, deletions count. Pure-deletion PRs get their own rungs rather than a leniency clause.
  • No codegen in new contract types. All new interface/runtime types are hand-written Dart 3 sealed classes and records — no freezed, no json_serializable. (The interface package's freezed convention has a measured ~10× generated-line multiplier that would blow PR budgets.)
  • Duplication between phases is fine. Old and new implementations may coexist; the old path keeps running production until a single, small "flip" PR switches it.
  • Fidelity gate. The 1,224-line open_code_server_service_test.dart is the behavioral spec for process lifecycle. The extraction is proven by those scenarios passing against the new code (PR 10) — not by review alone. To make that possible, all behavioral changes (health pacing, record timing, restart) are explicit policy knobs that default to legacy behavior until the flip (PR 12).

1Target architecture (recap)

Full analysis lives in the session report; the contract in one breath:

BridgePluginDescriptor (const, inert)— start(PluginHost) → BridgePlugin { api, status, describe(), shutdown(budget) }

The three pieces

  • Descriptor — registration unit. Declares id, CLI options, validateConfig (runs before the startup mutex and before any bridge gets replaced), and start(host). Replaces the BridgePluginFactory typedef (bridge_runtime_runner.dart:67).
  • BridgePlugin — a live instance: the existing BridgePluginApi (unchanged, must stay the same object for the plugin's lifetime), a replay-latest sealed PluginStatus stream with a legal-transition state machine (no Failed after Stopping), describe() diagnostics, and ordered shutdown(budget). dispose() and shutdown() are contractually idempotent (the orchestrator keeps calling dispose() until PR 12, so double-dispose is a supported state during migration).
  • PluginHost — opt-in services: config, per-plugin state dir, bridge identity facts (incl. isLiveBridgeProcess capability for stale-kill authorization), process spawn with stdin (ACP needs it), port probing, atomic JSON store with locked update(), clock + environment seams for tests.

Archetype fit & selective start

  • Managed runtime (OpenCode): uses the new ManagedProcessSupervisor from bridge/sesori_plugin_runtime — extracted, reusable lifecycle skeleton (stale cleanup, port selection, spawn, health, ownership, stop).
  • Direct CLI / remote server: a SteadyPluginLifecycle mixin reduces lifecycle to ~5 lines; no process machinery, no ceremony.
  • Registered ≠ started. Descriptors are const and side-effect free. The bridge starts only the enabled subset (CLI --plugin / settings), never "all registered". Every started plugin is individually stoppable via shutdown() — the PluginManager (PR 5) owns start/stop per id. Honest scope: the manager API is N-ready, but today's orchestrator bakes a single BridgePluginApi into ~10 constructor-injected components (bridge_runtime.dart:60-162) — running N plugins concurrently, or stopping/starting a plugin mid-session, requires an orchestrator/DI rebind that belongs to the protocol-portability track, not this ladder. stopPlugin on the active plugin while a session runs therefore has defined semantics now (PR 5): it cancels the session first.
Out of scope (separate track): protocol portability — the untyped Map<String,dynamic> payloads in BridgeSseEvent, the SSEManager._toOpenCodeFormat mobile wire envelope, the projectId == projectPath assumption in worktree_cleanup.dart, and the orchestrator/DI rebind needed for multiple concurrently active plugins. Lifecycle alone does not make a second plugin shippable end-to-end; that track needs mobile coordination and is deliberately not in this ladder.

PR 2 Lifecycle contract types (pure additions)

Goal: land the entire new contract in sesori_plugin_interface with nothing consuming it yet, so the contract itself gets a focused review.

  • BridgePluginDescriptor (id, displayName, options, validateConfig, start(PluginHost)), PluginOption (with a typed-parse hook so plugins don't re-implement "int.parse + usage error"), PluginConfig.
  • BridgePlugin (api, status, currentStatus, describe(), shutdown({budget})) and the sealed PluginStatus family: Starting / Ready / Degraded / Restarting / Failed / Stopping / Stopped. Degraded carries since, recoverable, and requiresUserAction + hint (remote-plugin auth expiry). The legal-transition state machine is documented on the type and enforced by the mixin: no Failed after Stopping; stream closes after Stopped.
  • PluginHost + service interfaces: HostProcessService (spawn returns SpawnedProcess with stdin, identity capture, exitCode; inspect/list/signalGraceful/signalForce; isLiveBridgeProcess classification via BridgeHostInfo), HostPortService.isBindable, HostJsonStore (atomic read/write/delete/quarantine plus locked update()), clock, environment, startAborted (cooperative abort signal).

Phase B — Seam flip

PR 4 Seam flip via LegacyOpenCodeDescriptor riskiest PR — keep it surgical

Goal: the runner consumes the new contract while zero OpenCode code moves. A bridge-app-internal descriptor wraps today's exact flow.

  • LegacyOpenCodeDescriptor (in bridge/app): options = the four existing flags (identical names/help/defaults); validateConfig = the --no-auto-start-requires---port rule; start(host) = today's resolveServer + OpenCodeServerService flow + OpenCodePlugin(serverUrl, password) construction, returning a BridgePlugin wrapper whose shutdown() runs dispose-then-stopOwnedServer in order and whose describe() reports the endpoint.
  • The wrapper's shutdown() is guarded/idempotent, and a test covers the double-dispose path: the orchestrator's finally still calls _plugin.dispose() (orchestrator.dart:359) until PR 12, so dispose-then-shutdown must be safe by contract, not by accident.
  • Runner changes (bridge_runtime_runner.dart): the selected descriptor's option-level validate hooks (on present, non-empty values) and validateConfig both run at arg-parse time, strictly before the mutex and before enforceSingleLiveBridge (a typo'd flag must never terminate a healthy resident bridge); PluginConfigException maps to the usage-error flow; mutex → enforce-single-bridge → build host → descriptor.start(host); the mutex is held until start() settles — no Future.timeout. A runner-initiated abort surfaces as PluginStartAbortedException from start() and is handled as "aborted as requested", not as the error-exit path.
  • Ordered shutdown: await plugin.shutdown(budget) as an ordered step before BridgeShutdownCoordinator's parallel phase; the coordinator's process-wide backstop timer is armed before the ordered phase, sized budget + slack (no window with no timer running), and its exit code is derived from the terminal-failure latch (exit 1 if PluginFailed was recorded — a hung shutdown after failure must not exit 0). registerOwnedOpenCodeShutdown folds into the wrapper's shutdown().
  • Failure surfacing: terminal-failure latch — first PluginFailed recorded immediately after start() returns, checked right before session.run(), late-bound to session.cancel() after. Honest delta note: the latch machinery lands here, but nothing can emit PluginFailed after a successful start until the exit monitor activates in PR 12 — start-time failures already exit 1 today.
  • Delete the BridgePluginFactory typedef and _createOpenCodePlugin. BridgeConfig.serverURL/serverPassword stay for now (fed from the legacy flow) so the orchestrator is untouched. Update bridge_runtime_runner_test.dart / bridge_runtime_server_test.dart to the new seam.
Why always-green: the descriptor delegates to byte-identical existing services; CLI surface and ownership file are unchanged. Deliberate deltas: shutdown becomes ordered (was: parallel race between SSE teardown and OpenCode kill) and the backstop exit code becomes latch-aware. Manual smoke: normal start, --no-auto-start attach, second-bridge replacement prompt, Ctrl-C shutdown, kill-opencode-mid-session.

PR 5 Plugin selection, enablement & PluginManager

Goal: the capability you asked for explicitly — registered plugins are not auto-started; each is independently startable/stoppable.

  • PluginManager in bridge/app: holds the descriptor registry, startPlugin(id) / stopPlugin(id) (per-id, idempotent, each stop honoring that plugin's shutdown(budget)), activePlugins view. Runner uses it for startup/shutdown.
  • Defined stop semantics: stopPlugin on the active plugin while a session is live first cancels the session (session.cancel()), then runs shutdown() — never leaves a zombie bridge serving requests against a stopped plugin. (The orchestrator subscription today only logs "plugin event stream closed"; that is not an acceptable end state.)
  • Selection policy: --plugin <id> CLI flag + enabledPlugins in bridge settings (BridgeSettingsRepository); default = opencode when unset, so existing installs see zero change. Only the selected descriptor's options are registered into the ArgParser (two-pass parse: --plugin first, then full parser; documented --help behavior).
  • Current constraint enforced with a clear error: exactly one active plugin. Scope honesty: the manager API is N-ready, but multi-active (and mid-session start/stop without cancel) needs the orchestrator/DI rebind tracked in the companion protocol-portability work — a single BridgePluginApi is constructor-baked into ~10 components today.
Why always-green: defaults reproduce today's behavior exactly; new flags are additive.

PR 8 Supervisor start / attach — with legacy-default policy knobs

Goal: the start path, hardened semantics available but defaulting to legacy behavior.

  • ManagedRuntimeSpec (executable, args/env builders, structured probeHealth returning {healthy, version, detail}, optional validateRuntime, port policy, budgets) with two explicit policy knobs:
    • healthPolicy: attemptCount(n, delay) (legacy: 5 × 500 ms) or deadline(d) (new default at flip: 30 s) — the old test suite hard-asserts exact delay sequences, so pacing must be configurable, not replaced;
    • recordTiming: afterSpawn (legacy: record written only once a real pid exists — the frozen schema's openCodePid is required non-null, so there is no representable pre-spawn state) or intentSideFile (new: see PR 9).
  • Start sequence: explicit-port path pre-probes bindability and diagnoses the occupant via host.processes ("port 4096 held by pid N: …"); spawn-failure (ENOENT) fails fast and skips port retries; child-exit-before-first-healthy-probe is authoritative failure regardless of probe success (squatter defense).
  • Cooperative abort: supervisor checks host.startAborted at every phase boundary and rolls back (kill child, delete record), then settles by throwing PluginStartAbortedException — pairs with the runner holding the mutex until settlement and distinguishing abort from start failure by type.
  • attach() mode (today's --no-auto-start): probe, never own, never kill; degraded-until-reachable semantics. Port the start/attach/port/health scenarios from the legacy suite.
Why always-green: still no production consumer. Restart/exit-monitoring deliberately NOT in this PR (see PR 9) to keep it under budget.

PR 9 Supervisor restart, exit monitoring, port pinning, intent side-file all default-off

  • Exit monitoring + stderr logging with a [runtimeId] prefix (reinstates what dead server/process.dart dropped); bounded restart-with-backoff emitting PluginRestarting; port pinned across restarts (the plugin's HTTP/SSE stack is address-frozen; if the port never frees → PluginFailed); restarts disarmed during stop() so clean shutdowns can't emit Failed.
  • Intent record goes to a bridge-private side file (e.g. <stateDir>/<runtimeId>-start-intent.json), written before spawn and resolved after — never into the frozen opencode-processes.json, whose required non-null openCodePid cannot represent "no pid yet" and whose readers in old bridges must never see a placeholder. A dedicated mixed-version test proves a pre-migration bridge reading the runtime directory is unaffected by the side file's presence.
  • All features here are off by default and activate only when the real descriptor opts in at PR 12 — PR 10's wrapper keeps them off.
Why always-green: no production consumer; everything default-off.
